The Mailbox Trick

Jun 2026 4 min read

Today I did something that felt like cheating. I added three new secrets to a project's encrypted env file — secrets a teammate would later read on his machine — without ever holding the key to decrypt anything in that file. Not the new values. Not the old ones. Nothing.

That's the moment I finally understood, in my bones, what asymmetric encryption actually does. And I want to write it down because the mental model that clicked for me is embarrassingly simple.

Two keys, not one

The tool I was using is called dotenvx. It encrypts the values in a .env file so the file itself can be safely committed to git. Anyone can read the repo. Nobody (without the right key) can read the secrets.

The trick is that encryption and decryption use different keys:

  • Public key — sits in .env.local, committed to git, visible to literally anyone on the internet. Used only to lock things up.
  • Private key — sits in .env.keys, never committed, shared only between teammates. Used only to unlock things.

They're mathematically related, but you can't derive one from the other in any reasonable amount of time. The public key locks the box. Only the private key opens it.

The mailbox

Here's the analogy that finally made it click:

Imagine a mailbox bolted to a wall in public. There's a slot on the front. Anyone walking past can drop a letter in. No ID required, no questions asked. The slot only goes one way — once your letter is inside, you can't fish it back out.

Around the back, there's a locked door. Only the postman has the key. He's the only person who can open it up, pull the letters out, and read them.

That's it. That's the whole thing.

  • The mailbox slot is the public key.
  • The back-door key is the private key.
Locking is public. Unlocking is private. Two completely separate powers, on two completely separate keys.

What this means in practice

I was working on a branch. I needed to add some new config values into the encrypted env file. I didn't have my teammate's private key. I'd never had it. I never needed it.

I just walked up to the mailbox, dropped my letters into the public slot, and committed the result. Git delivered the sealed envelopes. My teammate pulled the branch, and the private key he'd been holding since day one opened them right up.

No coordination. No Slack thread saying "hey, can you share the new value with me?" No password manager hand-off. Just git pull, and his existing setup did the rest.

Why this is quietly genius

Once you see it, the implications are everywhere:

  • A brand new contributor can add secrets to a repo on day one — they just need the public key, which is already in the file.
  • A CI bot or background agent can encrypt new values without ever holding decryption power. If it gets compromised, it can lock things up, but it can't read anything.
  • Rotating the private key is a deliberate event. Adding new encrypted values is not — anyone can do it.

The genius of the design is that it separates the authority to write secrets from the authority to read them. Most security systems collapse those two powers into one credential. This one doesn't. Which means the credential you protect with your life (the private key) gets shared as rarely as possible, while the credential that gets handled by anything and everything (the public key) is harmless if leaked. Because it already is leaked. That's the point.

How to spot it yourself

If you open an encrypted env file, you'll see lines like:

NEXT_PUBLIC_API_URL=encrypted:BCkLTy/Vg+CV...

The encrypted: prefix marks values that were sealed with the public key. To anyone without the private key, the rest is gibberish. To anyone with it, it's a real URL, a real token, a real secret — restored at runtime, used to boot the app, then thrown away.

That little prefix is doing a surprising amount of work. It's the visible edge of an idea that took mathematicians a long time to find and engineers a long time to get comfortable with: the idea that you can give the whole world the power to send you a secret, while keeping the power to read it entirely to yourself.

I added three letters to a mailbox today. I never held the key. The system worked anyway. That's the whole story.
